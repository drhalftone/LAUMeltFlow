# GNN Surrogate for Bead Chain Physics

Two approaches to learning bead chain dynamics with a GNN:

## Approach 1: Direct Physics (per-bead MLP, no SHAKE)

Learns the full single-step physics (spring + damping + gravity + drag + symplectic Euler) directly from volume-sampled training data. No constraint projection needed — just stiff springs.

### Architecture

Each bead runs the same MLP with shared weights:

```
 LEFT NEIGHBOR                      SELF                       RIGHT NEIGHBOR
 [dpos_x, dpos_y,              [vel_x, vel_y,                [dpos_x, dpos_y,
  dvel_x, dvel_y,               mass,                         dvel_x, dvel_y,
  mass,                          is_fixed]                     mass,
  rest_len]                        (4)                         rest_len]
     (6)                           |                              (6)
      |                            |                               |
      +------------+---------------+---------------+---------------+
                   |       CONCATENATE (16)        |
                   +---------------+---------------+
                                   |
                         ResMLPBlock x3
                        (LayerNorm + GELU
                        + residual connection)
                                   |
                          Linear H -> 4
                                   |
                          [d_pos_x, d_pos_y,
                           d_vel_x, d_vel_y]
                                (4)
                                   |
                    pos_new = self_pos + d_pos
                    vel_new = self_vel + d_vel
```

All features are **relative**: neighbor positions and velocities are expressed
as differences from the current bead's state. Self position is omitted
(implicitly zero in the local frame), making the network translation-invariant.

Ghost beads (missing neighbors at endpoints) have all-zero features:
d_pos=0, d_vel=0, mass=0, rest_len=0.

### Quick Start

#### 1. Generate volume-sampled training data

```bash
python generate_volume_data.py --n_samples 2000000
```

Generates 2M samples: half full-volume, half near-zero. The near-zero
samples use a narrow Gaussian around the rest state (small velocities,
rods near rest length) to ensure the model is accurate at the initial
state and low-energy configurations where the simulation spends most of
its time.

Neighbor positions are sampled in **polar coordinates** (r, theta) around
the bead, with r constrained near the rest length. This ensures
physically realistic neighbor distances.

Volume bounds (from Qt simulation analysis):
- Self velocity: vel_x [-13.1, 13.1], vel_y [-7.8, 7.8]
- Neighbor distance: r = rest_length * [0.95, 1.05], theta = [0, 2pi]
- Neighbor relative velocity: dvx [-26.2, 26.2], dvy [-15.6, 15.6]
- Near-zero scale: 0.05 (5% of full range, Gaussian)

#### 2. Train

```bash
python bead_train.py --epochs 200 --hidden_dim 64
```

Or double-click `bead_train.bat` to run in a separate window.

Training uses mixed precision (AMP) on GPU for speed. With an RTX 4070 Ti
SUPER, 200 epochs on 2M samples takes ~10 minutes.

#### 3. Visualize

```bash
python bead_visualize.py --model_path outputs/bead_best.pt
```

Shows ground truth (stiff springs) vs GNN side by side with trajectory
error plot.

#### 4. Analyze bounds (optional)

```bash
python analyze_bounds.py ../qt/bead_training.csv
```

Reads the Qt bead training CSV and prints per-feature min/max to verify
or update the volume bounds.

### Training Data

Each training sample is one bead at one timestep: given its current state
and its neighbors' states, what are the position and velocity deltas after
one integration step? The exact output is computed by running the bead
physics formula (spring + damping + gravity + drag + symplectic Euler) —
no simulation needed, just a direct function evaluation.

The challenge is **where to sample inputs**. Simulation trajectories only
visit a thin manifold in the full state space (correlated positions,
velocities, and stretches). If the GNN only trains on trajectory data, it
fails on states outside that manifold — including states caused by its own
prediction errors during rollout. But if it only trains on uniform random
states, it under-represents the states that actually matter.

The solution is three populations of training data:

| Population | Share | Sampling | Purpose |
|------------|-------|----------|---------|
| Full-volume | 40% | Uniform random across bounds | Cover the full reachable space, robustness to out-of-distribution states |
| Near-zero | 40% | Gaussian(0, 5% of range) | Accurate at rest state, initial conditions, low-energy configurations |
| Trajectory | 20% | Actual simulation runs | Cover the correlated states that occur during real chain dynamics |

**Full-volume samples** are generated by independently sampling each input
feature uniformly within bounds discovered from Qt simulation analysis.
Neighbor positions use polar coordinates (r near rest length, uniform angle)
to ensure physically realistic distances. These samples include many
physically impossible combinations (e.g., high velocity + zero stretch)
which helps the model handle error states gracefully.

**Near-zero samples** use a narrow Gaussian centered at zero for velocities,
stretch, and relative velocities. The initial chain state (horizontal, all
velocities zero, all rods at exact rest length) sits at the center of this
distribution. Without these, the model produces spurious forces at t=0.

**Trajectory samples** come from running 12 actual simulations (no SHAKE,
stiff springs only) with random initial velocity perturbations, recording
per-bead input/output pairs at every timestep. These capture the real
correlations between bead states — e.g., when the chain swings down, tip
beads move fast while anchor-adjacent beads move slowly, and rod stretches
follow specific patterns. The model needs to be especially accurate on
this manifold since that's where it operates during inference.

### Training Data Format

Single `volume_data.npz` with:
- `X`: (N, 16) input features per bead sample
- `Y`: (N, 4) target deltas [d_pos_x, d_pos_y, d_vel_x, d_vel_y]
- `bead_ids`: (N,) which bead index (0-15)
- Physics constants: stiffness, damping, drag, gravity, dt, rest_length

Input features (16):

| # | Feature | Description |
|---|---------|-------------|
| 0 | vel_x | Self velocity x |
| 1 | vel_y | Self velocity y |
| 2 | mass | Self mass |
| 3 | is_fixed | 1 if anchor bead, 0 otherwise |
| 4 | l_dpos_x | Left neighbor relative position x |
| 5 | l_dpos_y | Left neighbor relative position y |
| 6 | l_dvel_x | Left neighbor relative velocity x |
| 7 | l_dvel_y | Left neighbor relative velocity y |
| 8 | l_mass | Left neighbor mass (0 for ghost) |
| 9 | l_rest | Left rod rest length (0 for ghost) |
| 10 | r_dpos_x | Right neighbor relative position x |
| 11 | r_dpos_y | Right neighbor relative position y |
| 12 | r_dvel_x | Right neighbor relative velocity x |
| 13 | r_dvel_y | Right neighbor relative velocity y |
| 14 | r_mass | Right neighbor mass (0 for ghost) |
| 15 | r_rest | Right rod rest length (0 for ghost) |

### Files

| File | Purpose |
|------|---------|
| `bead_model.py` | `BeadGNN` — shared-weight ResMLPBlock model with `step_chain()` for inference |
| `bead_train.py` | Training loop with AMP, normalization, cosine schedule |
| `bead_train.bat` | Run training in a separate command window |
| `bead_visualize.py` | Live side-by-side animation (ground truth vs GNN) |
| `generate_volume_data.py` | Volume + near-zero data generation |
| `analyze_bounds.py` | Discover state space bounds from Qt CSV export |

## Approach 2: U-Net GNN (SHAKE corrections)

Learns to replace the iterative SHAKE constraint solver with a single forward
pass through a binary tree. See the encode/decode architecture below.

### Quick Start

#### 1. Generate training data

From the `whip/` directory:

```bash
python generate_data.py --n_samples 100000
```

#### 2. Train

```bash
python train.py --epochs 500 --hidden_dim 64 --batch_size 32 --lr 1e-3
```

Or double-click `train.bat`.

#### 3. Evaluate

```bash
python evaluate.py --model_path outputs/best_model.pt --n_steps 30000
```

### Architecture

Binary tree U-Net with encode/decode passes and skip connections:

```
ENCODE (up-pass):
  Level 0: bead MLP([self | left | right]) -> h    (16 beads, save skip)
  Level 1: MLP([left_child | right_child]) -> h    (8 nodes, save skip)
  Level 2: MLP([left_child | right_child]) -> h    (4 nodes, save skip)
  Level 3: MLP([left_child | right_child]) -> h    (2 nodes, save skip)
  Level 4: MLP([left_child | right_child]) -> h    (1 root, bottleneck)

DECODE (down-pass):
  Level 3: MLP([parent_output | skip]) -> d         (2 nodes)
  Level 2: MLP([parent_output | skip]) -> d         (4 nodes)
  Level 1: MLP([parent_output | skip]) -> d         (8 nodes)
  Level 0: MLP([parent_output | skip]) -> correction (16 beads, output)
```

All nodes at the same level share MLP weights.

### Training Data Format

Paired `.npy` files matched by numeric suffix:

```
data/
  input/               # (16, 7) per frame: [pos_x, pos_y, vel_x, vel_y, mass, type, level]
    frame000000.npy
    ...
  target/              # (16, 4) per frame: [dx, dy, dvx, dvy] SHAKE corrections
    frame000000.npy
    ...
  graph.npz            # shared graph connectivity
```
